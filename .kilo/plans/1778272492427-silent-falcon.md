# Plan: `awsiotweb` ‚Äî AWS-Native IoT Dashboard for GPIO Orchestration

## 0. Context & Scope

This plan adds a new sibling directory `awsiotweb/` to the existing `awsiot` repo (which currently targets ESP32-C3 firmware via PlatformIO). The new subproject is a React + TypeScript web dashboard that controls GPIO states on **ESP32-C3 edge devices** through AWS IoT Core. The existing PlatformIO firmware under `src/` will be extended to implement named device shadow (`gpio`) synchronization; the Raspberry Pi path mentioned in the prompt is explicitly **out of scope** per user decision.

Repo layout after completion:

**Open questions (confirm before implementation):**
1. IaC preference: **AWS CDK (TypeScript)** vs **Amplify Gen 2**. Plan defaults to **CDK** for precise IAM/IoT policy control, with Amplify Libraries v6 used *only* on the frontend. Confirm.

## 1. Technology Stack

### Frontend (`awsiotweb/app`)
| Concern | Choice | Rationale |
|---|---|---|
| Framework | **React 18 + TypeScript + Vite** | Fast HMR, ESM, tree-shakable AWS SDK |
| Routing | **React Router v6** | Declarative nested routes for device detail views |
| Server state | **TanStack Query v5** | Caching, retry, optimistic mutations, suspense |
| Client state | **Zustand** | Small footprint for MQTT connection + UI state |
| AWS Auth | **AWS Amplify v6 (`aws-amplify/auth`)** | Cognito Hosted UI, token refresh, `fetchAuthSession` for SigV4 creds |
| IoT transport | **AWS IoT Device SDK v2 JS (`aws-iot-device-sdk-v2`) over MQTT-WSS with SigV4** using Cognito Identity Pool credentials | Avoids shipping X.509 to browsers; aligns with least-privilege scoped IAM |
| UI kit | **shadcn/ui + Tailwind CSS** | Accessible primitives, theming |
| Forms / validation | **react-hook-form + Zod** | Type-safe payload validation before publish |
| Charts (telemetry) | **Recharts** | Lightweight timeseries |
| Testing | **Vitest + React Testing Library + Playwright** | Unit + E2E |
| Lint/format | **ESLint (typescript-eslint) + Prettier** | |

### Backend / Infra
- **AWS CDK v2 (TypeScript)** ‚Äî single app with stacks: `AuthStack`, `IotStack`, `ApiStack`, `WebStack`.
- **AWS Lambda (Node.js 20, TypeScript, esbuild)** behind **API Gateway HTTP API** for business logic (device registration, audit log, command validation).
- **DynamoDB** for device metadata (`Devices` table: PK `tenantId`, SK `deviceId`) and audit log (TTL-enabled).
- **AWS IoT Core**: Things, Thing Groups (per tenant), Device Shadow (Classic + Named `gpio`), Rules Engine (IoT ‚Üí Lambda for audit; IoT ‚Üí CloudWatch Logs for observability).
- **CloudFront + S3** for SPA hosting (OAC, no public bucket), **Route 53 + ACM** for custom domain (optional).

### Edge (existing `src/` firmware, extended)
- **ESP32-C3** via **PlatformIO + Arduino framework** (matches current `platformio.ini`).
- MQTT client: retain `PubSubClient` + `WiFiClientSecure` **or** migrate to **AWS IoT Device SDK for Embedded C v2** (`aws-iot-device-sdk-embedded-C`) via ESP-IDF component for first-class shadow support. Plan defaults to staying on Arduino + hand-rolled shadow JSON (simpler, matches current code) with **ArduinoJson** for parsing.
- Persistent state in NVS (`Preferences` library) for last-known GPIO values across reboots.
- Watchdog task (FreeRTOS `esp_task_wdt`) to force reboot on stuck MQTT loop.

**Key design choices:**
- The frontend uses **two channels simultaneously**:
  - **HTTPS ‚Üí API Gateway ‚Üí Lambda** for *writes* (command validation, audit trail, rate limiting).
  - **MQTT-WSS (SigV4)** *direct subscribe* to `$aws/things/{deviceId}/shadow/name/gpio/update/documents` and `.../accepted` for *real-time reads*. No Lambda in the read path ‚Äî eliminates fan-out latency.
- Command path goes through Lambda (not direct browser‚ÜíIoT publish) so we can enforce cross-tenant checks, write audit entries, and keep IoT policy narrow.
- **Named shadow `gpio`** isolates GPIO state from classic shadow (which may hold telemetry/health).

### Shadow document schema (`name=gpio`)
ESP32-C3 GPIO numbering (safe pins for devkit-m-1: GPIO 0‚Äì10, 18‚Äì21, avoiding strapping pins where possible):
```json
{
  "state": {
    "desired":  { "pins": { "2": "HIGH", "4": "LOW", "5": "HIGH" } },
    "reported": { "pins": { "2": "HIGH", "4": "LOW", "5": "HIGH" },
                  "fw": "1.2.0", "rssi": -58, "ts": 1736200000 }
  }
}
```

### MQTT topic plan
| Topic | Direction | Purpose |
|---|---|---|
| `$aws/things/{id}/shadow/name/gpio/update` | device‚Üícloud, cloud‚Üídevice | command + ack |
| `$aws/things/{id}/shadow/name/gpio/update/delta` | IoT‚Üídevice | applied by agent |
| `$aws/things/{id}/shadow/name/gpio/update/documents` | IoT‚Üíbrowser | reconcile UI |
| `devices/{tenantId}/{id}/telemetry` | device‚Üícloud | metrics (optional) |
| `devices/{tenantId}/{id}/events` | device‚Üícloud | lifecycle events |

## 3. Security & Identity

### Cognito
- **User Pool** with:
  - MFA optional (TOTP), password policy ‚â•12 chars.
  - Custom attribute `custom:tenantId` (immutable, required).
  - Groups: `Admin`, `Operator`, `Viewer` mapped to IAM roles via Identity Pool role mapping rules.
- **Identity Pool** federates the User Pool ‚Üí three authenticated IAM roles (`AdminRole`, `OperatorRole`, `ViewerRole`). No unauthenticated access.

### IAM policy scoping (browser credentials)
Uses Cognito policy variables (`${cognito-identity.amazonaws.com:sub}`) and User Pool attribute passthrough (`${aws:PrincipalTag/tenantId}` via role session tags set in the Identity Pool role trust policy).

**OperatorRole IoT policy (browser side)** ‚Äî subscribe-only to own tenant shadows, publish only to a request topic that a Lambda relays:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow", "Action": "iot:Connect",
      "Resource": "arn:aws:iot:us-west-2:*:client/${cognito-identity.amazonaws.com:sub}" },
    { "Effect": "Allow", "Action": ["iot:Subscribe"],
      "Resource": "arn:aws:iot:us-west-2:*:topicfilter/$aws/things/${aws:PrincipalTag/tenantId}-*/shadow/name/gpio/*" },
    { "Effect": "Allow", "Action": ["iot:Receive"],
      "Resource": "arn:aws:iot:us-west-2:*:topic/$aws/things/${aws:PrincipalTag/tenantId}-*/shadow/name/gpio/*" }
  ]
}
```
> Convention: `deviceId` is prefixed with `tenantId-` to make topic-pattern ABAC feasible without per-user policies.

### Device identity
- **Per-device X.509** issued by AWS IoT, embedded at build time via existing `generate_certs_header.py` (produces a `certs.h` included by firmware). NVS-backed alternative for production. Long-term: **AWS IoT Fleet Provisioning by Claim** so each ESP32-C3 bootstraps a unique cert on first boot using a shared claim cert.
- **Per-device IoT policy** attached to cert, scoped by the policy variable `${iot:Connection.Thing.ThingName}`:
```json
{
  "Effect": "Allow",
  "Action": ["iot:Publish","iot:Subscribe","iot:Receive"],
  "Resource": [
    "arn:aws:iot:us-west-2:*:topic/$aws/things/${iot:Connection.Thing.ThingName}/shadow/*",
    "arn:aws:iot:us-west-2:*:topicfilter/$aws/things/${iot:Connection.Thing.ThingName}/shadow/*"
  ]
}
```

### Lambda authorization
- Lambda validates Cognito JWT (via API Gateway JWT authorizer), then cross-checks `custom:tenantId` claim against the `tenantId` prefix on the requested `deviceId`, rejecting mismatches with 403. Admin group bypass only with explicit allow-list check.

## 4. Implementation Lifecycle

### Phase 1 ‚Äî Repo scaffolding
1. Create `awsiotweb/` with `pnpm` workspaces: `app`, `infra`, `lambda`.
2. Root `package.json` with `turbo` pipelines (`dev`, `build`, `deploy`, `test`).
3. Add `.gitignore`, `.editorconfig`, `tsconfig.base.json`, shared ESLint/Prettier configs.

### Phase 2 ‚Äî Infrastructure as Code (CDK)
Stacks, in order:
1. **`AuthStack`**: Cognito User Pool, User Pool Client (PKCE, no secret), Identity Pool, three IAM roles with ABAC on `tenantId`, role mapping rules.
2. **`IotStack`**:
   - Thing Type `gpio-device`, Thing Group per tenant (created lazily by Lambda).
   - IoT Policy (device) with `${iot:Connection.Thing.ThingName}` scoping.
   - IoT Rules: `ShadowAuditRule` ‚Üí Lambda (`auditShadow`), `DeviceEventsRule` ‚Üí CloudWatch Logs.
   - Fleet Provisioning template + claim policy (Phase 6).
3. **`ApiStack`**:
   - DynamoDB `Devices` (PK `tenantId`, SK `deviceId`, GSI `byDeviceId`).
   - DynamoDB `AuditLog` (PK `tenantId`, SK `ts#requestId`, TTL 90d).
   - Lambdas: `registerDevice`, `listDevices`, `sendCommand`, `auditShadow`.
   - HTTP API with Cognito JWT authorizer, routes: `POST /devices`, `GET /devices`, `POST /devices/{id}/command`.
4. **`WebStack`**: S3 (private) + CloudFront distribution with OAC, SPA redirect (`403/404 ‚Üí /index.html`), optional ACM cert + Route 53 record.

### Phase 3 ‚Äî Device provisioning & shadows
1. CLI script `awsiotweb/infra/scripts/provision-device.ts` that, given `tenantId` and `deviceName`:
   - Creates Thing `{tenantId}-{deviceName}`, attaches to tenant Thing Group.
   - Creates keys+cert via `CreateKeysAndCertificate`, attaches device IoT policy.
   - Writes DynamoDB metadata row.
   - Emits a folder with `certificate.pem.crt`, `private.pem.key`, `AmazonRootCA1.pem`, and `config.json` (endpoint, thingName, shadowName=`gpio`) consumable by `generate_certs_header.py`.
2. Initialize named shadow `gpio` with empty desired/reported via `UpdateThingShadow` API call post-provisioning.

### Phase 4 ‚Äî ESP32-C3 firmware extension
Path: `src/shadow/` alongside existing `src/main.c`.
- Extend current MQTT connection (port 8883, MTLS via `WiFiClientSecure`) to additionally subscribe:
  - `$aws/things/{thingName}/shadow/name/gpio/get/accepted`
  - `$aws/things/{thingName}/shadow/name/gpio/update/delta`
- On boot (post Wi-Fi + NTP + MQTT connect): publish empty message to `‚Ä¶/shadow/name/gpio/get` to seed from cloud; apply `desired.pins` idempotently against GPIO via `pinMode`+`digitalWrite`; persist to NVS `Preferences`.
- On `/update/delta`: parse with ArduinoJson (`StaticJsonDocument<512>`), apply each pin, build `reported` payload (`{pins, fw=FIRMWARE_VERSION, rssi=WiFi.RSSI(), ts=time(nullptr)}`), publish to `‚Ä¶/shadow/name/gpio/update`.
- LWT: on `devices/{tenantId}/{thingName}/events` with `{"online":false,"ts":‚Ä¶}` retained=false, QoS 1; publish `{"online":true}` on connect.
- Reconnect with exponential backoff (1s‚Üí60s, ¬±20% jitter), `PubSubClient.setBufferSize(1024)` to fit shadow docs.
- Optional: periodic telemetry every 60s on `devices/{tenantId}/{thingName}/telemetry` (RSSI, heap, uptime).
- Watchdog: feed from main loop; reboot if no successful MQTT loop in 120s.

### Phase 5 ‚Äî Frontend implementation
Module layout:
```
app/src/

  auth/                   # Cognito hooks, guarded routes
  iot/
    mqttClient.ts         # Singleton SDK v2 over WSS+SigV4
```

**MQTT client (key behaviors):**
- Acquire credentials via `fetchAuthSession()`, build `AwsIotMqttConnectionConfigBuilder.new_builder_for_websocket().with_credentials_provider(...)`.
- Auto-refresh: listen to Amplify `Hub` auth events, rebuild connection on token refresh.
- Expose `subscribe(topic, qos, handler)` returning unsubscribe fn; multiplex subscriptions via internal ref-count.

**Optimistic UI (`useCommand`):**
- On mutate: write `desired.pins[pin]=HIGH` into TanStack Query cache keyed by `['shadow', deviceId]`, display pending spinner.
- On MQTT `/update/documents` receiving matching `reported`, reconcile; if divergent after timeout (e.g., 10s), mark error and rollback.
- Use `onError` to rollback cache to snapshot.

### Phase 6 ‚Äî Fleet provisioning & hardening (post-MVP)
1. Fleet Provisioning by Claim for ESP32-C3: shared claim cert flashed at factory, exchanged for unique per-device cert on first boot via `$aws/certificates/create/json` and `$aws/provisioning-templates/{template}/provision/json`; unique cert persisted to NVS. Claim policy restricted to those two topic patterns only.
2. CloudWatch dashboards: MQTT connect/disconnect, shadow update count, Lambda errors.
3. WAF on CloudFront; rate limit `POST /devices/{id}/command` via API Gateway usage plan or Lambda + DynamoDB token bucket.

### Phase 7 ‚Äî CI/CD
- GitHub Actions: `lint`, `test`, `cdk synth`, `cdk deploy` (OIDC to AWS, no long-lived keys), Playwright smoke against ephemeral preview stack.

## 5. Operational Excellence

### Async communication & offline handling
- **Shadows are source of truth** for GPIO state; UI never trusts its own optimistic value beyond reconciliation window.
- ESP32-C3 persists last-known `reported` in NVS; on reconnect publishes `/get` and reconciles against `desired` before applying further deltas. Deltas received while applying are queued in a small FreeRTOS queue.
- LWT + `events` topic drives a browser "offline" badge; Lambda `auditShadow` also flags stale `reported.ts` (>3√ó telemetry interval).

### Principle of Least Privilege
- Three distinct trust boundaries: **browser creds** (read-heavy, scoped by tenant tag), **Lambda execution role** (IoT `UpdateThingShadow` + DDB writes only on tenant-prefixed resources using IAM condition keys), **device cert** (thing-name-scoped).
- No `iot:*` wildcards anywhere. CDK `Grant` helpers only; explicit `resources` arrays; CDK-nag enabled in CI to fail on violations.
- Secrets: no secrets in Lambda env vars; config in SSM Parameter Store (SecureString).

### Resilience & error handling
- **Browser MQTT**: exponential backoff (SDK v2 built-in), surface status via Zustand store, show banner after >3s disconnect, automatic resubscribe on reconnect.
- **Token expiry**: intercept 401 from API Gateway with TanStack Query `retry` + Amplify `fetchAuthSession({ forceRefresh: true })`.
- **Lambda**: idempotency keys on `POST /command` (client-generated UUID stored in DDB with 5m TTL) to dedupe retries.
- **Device**: bounded FreeRTOS queue (drop-oldest), serial logs + optional MQTT log topic `devices/{tenantId}/{thingName}/logs` (rate-limited) forwarded via IoT Rule to CloudWatch; graceful shutdown publishes `{online:false}` on `events` topic before disconnect.
- **Observability**: X-Ray on Lambda + API GW; CloudWatch metric filters for `ShadowUpdateRejected`; alarms ‚Üí SNS.

### Testing strategy
- Unit: Lambda handlers with `aws-sdk-client-mock`; React hooks with RTL.
- Integration: ephemeral CDK stack per PR, Playwright drives real Cognito test user against an **ESP32-C3 simulator** (small Node script using `aws-iot-device-sdk-v2` that mimics shadow `get`/`update` semantics against AWS IoT Core).
- Chaos: inject disconnects in simulator, assert UI reconciles within SLO (‚â§5s).
